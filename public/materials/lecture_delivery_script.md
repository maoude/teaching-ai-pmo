# Lecture Delivery Script

## Module

From Plan to Control: Managing a Project Like a Real PMO

## Duration

180 minutes

## Instructor Opening

Today you are not only students. You are the PMO team responsible for delivering a Smart Campus Mobile App before the next semester. You have 12 weeks, a limited budget, a small team, and a sponsor who expects a demo by week 6.

Your task is to manage the project professionally. That means defining the scope, building the WBS, planning the schedule, tracking performance, identifying risks, and proposing corrective actions when the project starts to fail.

## Segment 1: Opening Scenario and Poll

### Time

0-15 minutes

### Say

The university president has approved the Smart Campus Mobile App. The sponsor wants quick progress and asks the team to start immediately. Before you do anything else, what should the project manager do first?

### Poll Question

What should the PM do first?

A. Start coding immediately  
B. Define scope and stakeholders  
C. Buy software tools  
D. Ask for more budget  

Correct answer: B.

### Debrief

Starting without scope creates confusion. A project manager first clarifies stakeholders, scope, constraints, assumptions, and success criteria. Without these, the team cannot know what success means.

## Segment 2: Planning and Charter

### Time

15-45 minutes

### Say

A project charter is not a long document for bureaucracy. It is the first agreement about why the project exists, what it should deliver, who cares about it, and what constraints must be respected.

### Student Task

Each group identifies:

1. Five stakeholders
2. Three measurable success criteria
3. Main constraints
4. One-sentence project vision

### Expected Stakeholders

- University president
- Students
- Faculty
- IT department
- Registrar
- Security or admin office

### Expected Success Criteria

- Pilot ready by week 10
- At least 80 percent of pilot users can log in successfully
- Critical bugs resolved before launch

## Segment 3: WBS Builder

### Time

45-80 minutes

### Say

A WBS is not just a task list. A good WBS is deliverable-oriented. It breaks the project into outputs that must exist for the project to be complete.

### Student Task

Groups build a three-level WBS.

### Model Structure

1. Smart Campus App
   1.1 Project Management
   1.2 Requirements
   1.3 Design
   1.4 Development
   1.5 Testing
   1.6 Deployment

### Debrief

If your WBS only says "code app" or "test app", it is too vague. A PMO needs enough structure to estimate, assign, track, and control the project.

## Segment 4: Schedule Rescue

### Time

90-120 minutes

### Say

The first schedule takes 15 weeks, but the sponsor requires 12 weeks. The PM cannot simply change numbers. The PM must change the plan.

### Student Task

Groups propose a 12-week plan using realistic schedule compression.

### Acceptable Actions

- Parallelize UI and backend work
- Reduce non-critical scope
- Delay advanced features
- Add temporary development support
- Automate repeated tests

### Warning

Removing testing is not optimization. It only hides risk until later.

## Segment 5: Dashboard Detective

### Time

120-155 minutes

### Week 6 Data

| Metric | Planned | Actual |
| --- | ---: | ---: |
| Progress | 50% | 38% |
| Cost | 20,000 USD | 24,000 USD |
| Critical bugs | 0 | 12 |
| Completed modules | 4 | 2 |

### Say

The dashboard is telling a story. The project is not only late. It is late, over budget, and technically unstable.

### Student Questions

1. Is the project healthy?
2. Which indicator shows schedule pressure?
3. Which indicator shows cost pressure?
4. Which indicator shows technical instability?
5. What should the PM communicate to the sponsor?

### Expected Diagnosis

The project is behind schedule because actual progress is below planned progress. It is over budget because actual cost is higher than planned cost. It is technically unstable because critical bugs are high and completed modules are fewer than expected.

## Segment 6: Week 8 Crisis

### Time

155-175 minutes

### Say

At week 8, database integration fails. QA reports many bugs. The sponsor refuses to extend the deadline. The team is overloaded.

What is the most professional recovery strategy?

A. Add more features to impress the sponsor  
B. Freeze scope and focus on login, schedule, and notifications  
C. Stop testing to save time  
D. Hide the delay until week 12  

Correct answer: B.

### Student Output

Each group prepares a three-minute PMO recovery briefing:

1. What happened?
2. What is the impact?
3. What corrective action is proposed?
4. What decision is needed from the sponsor?

## Closing

Project control is not about producing dashboards. It is about using information to make responsible decisions. A project manager must communicate problems early, protect critical deliverables, and propose realistic corrective action.
